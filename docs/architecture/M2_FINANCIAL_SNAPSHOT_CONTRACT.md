# M2-6 ‚Äî Household Financial Snapshot Contract (the Financial Kernel)

> **Status:** Accepted (the **canonical financial contract** of Life Capital OS V2; extends the
> household-scoped, multi-currency, immutable-snapshot M2 pattern and adds **ADR-012**). **Slice:** M2-6.
> **Depends on:** M2-2 (accounts), M2-3 (net worth + immutable snapshots + `FxService`), M2-4 (cashflow/budget),
> M2-5 (debt). **Consumed by:** every future module (M3+). Follows the
> [M2 architecture reference](./M2_HOUSEHOLD_WEALTH_ARCHITECTURE.md) + ADRs.

This document is the **single source of truth** for what a Financial Snapshot is. It is a **contract**: once a
consumer depends on `schemaVersion` _N_, that shape is frozen and only extended additively. Read ¬ß8 and ¬ß12
before changing anything here.

---

## 1. Purpose

A **Financial Snapshot** is an **immutable, point-in-time capture of a household's complete financial position**,
consolidated into the household base currency and stamped with the exact engine/FX/schema versions that produced
it. It is the **read model** for the whole product: net worth, assets, liabilities, debt, cashflow, budget,
allocation, currency exposure, and entity/household equity ‚Äî reconciled into **one row** with a stable shape.

It is the **single immutable financial truth** because:

- **One reconciled view.** M2-2‚Ä¶M2-5 each own a slice (accounts, net worth, cashflow, debt). Every consumer that
  needs "the household's finances" would otherwise re-query and re-reconcile four engines with subtly different
  assumptions. The snapshot reconciles them **once**, at capture, and freezes the result.
- **Reproducibility.** A report, score, or AI answer generated from snapshot `X` can be regenerated identically
  years later ‚Äî the inputs are frozen and the producing versions are recorded (¬ß2). Live tables drift; snapshots
  do not.
- **Decoupling.** Consumers depend on the **contract**, not on M2-2‚Ä¶M2-5 internals. Those engines can evolve
  (new columns, better FX) without breaking a single downstream module, because old snapshots keep their old
  shape and new snapshots advance `schemaVersion` additively.
- **Auditability.** Each snapshot records who generated it, from what, and a content hash ‚Äî a tamper-evident
  financial record for compliance (DPDP, firm trust).

**In scope (M2-6):** the `FinancialSnapshot` contract + table, a `FinancialSnapshotService` that **composes**
M2-2‚Ä¶M2-5 (never re-aggregates raw tables), immutable capture + versioned read APIs, timeline, a minimal
verification UI, and this contract doc. **Out of scope:** scheduled capture (M0 worker), point-in-time _write_
restore, and every consumer module in ¬ß10 (they arrive later and consume this).

## 2. Snapshot Contract (envelope)

Every snapshot carries a stable **envelope** (metadata) around its **payload** (¬ß3). The envelope never shrinks;
fields are only added.

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | `cuid` | **Snapshot ID** ‚Äî immutable primary key. |
| `householdId` | `cuid` | **Household ID** ‚Äî aggregate root (ADR-001); the scope of the snapshot. |
| `entityId` | `cuid?` | **Entity ID** ‚Äî **reserved** for a future per-entity snapshot; **always `null` in v1** (captures are household-level). Per-entity detail is already in `entityHoldings` (¬ß3); a per-entity _top-level_ snapshot is a forward-compatible extension that needs no reshaping. |
| `capturedAt` | `timestamp` | **Snapshot Timestamp** ‚Äî the instant the position was frozen (UTC). |
| `snapshotVersion` | `int` | **Snapshot Version** ‚Äî monotonic per `(householdId, entityId)`; 1, 2, 3‚Ä¶ (ordering/"latest"). |
| `schemaVersion` | `int` | **Schema Version** ‚Äî the version of the **payload shape** (¬ß3). Starts at **1**. Consumers pin to this. |
| `engineVersion` | `string` | **Engine Version** ‚Äî semver of the composing service + `@lcos/core` finance logic (e.g. `"m2-6.1.0"`). |
| `fxVersion` | `string` | **FX Version** ‚Äî identifier of the FX rate set used (e.g. `"static-2026-07"`); makes conversions reproducible. |
| `currency` | `ISO 4217` | **Currency** ‚Äî the household base currency the payload is consolidated into. |
| `provenance` | `json` | **Provenance** ‚Äî the source snapshot ids/counts the payload was composed from (e.g. `{ netWorthSnapshotId, accountCount, txCount, debtCount }`). |
| `generatedBy` | `enum` | **Generated By** ‚Äî `manual` \| `scheduled` \| `event` \| `migration` (who/what triggered capture). |
| `createdById` | `cuid?` | acting user id when `generatedBy = manual`. |
| `checksum` | `string` | **Checksum / Hash** ‚Äî SHA-256 over the canonical-JSON payload; detects tampering / accidental mutation. |
| `status` | `enum` | **Status** ‚Äî `active` \| `superseded` \| `void`. `active` is the trustworthy record; `void` marks a mistaken capture (a **new** snapshot corrects it ‚Äî the row is never edited). |
| `firmId` | `cuid` | tenant scope (indexed). |
| `payload` | `json` | the **¬ß3 contents** at `schemaVersion`. |

**Rule:** the envelope + `checksum` are computed at capture and **never recomputed**. `checksum` is verified on
read (optionally) but never regenerated in place ‚Äî a mismatch is surfaced, not "fixed".

## 3. Snapshot Contents (payload ‚Äî `schemaVersion = 1`)

The payload is a single JSON document with a **fixed top-level shape**. All monetary values are integer **minor
units** in the snapshot `currency`. Composed from the existing engines (source noted):

```jsonc
{
  "netWorth": {                 // M2-3 (computed at capture; also linked via provenance)
    "assetsMinor": 0,
    "liabilitiesMinor": 0,      // from liability Accounts (M2-3 basis; see ¬ßnote + ADR-012)
    "netWorthMinor": 0,
    "solvencyRatio": 0
  },
  "assets": [                   // M2-2 accounts where !isLiability, converted to base
    { "accountId": "...", "name": "...", "assetClass": "equity", "entityId": "...",
      "nativeCurrency": "USD", "nativeBalanceMinor": 0, "baseBalanceMinor": 0 }
  ],
  "liabilities": [              // M2-2 accounts where isLiability, converted to base
    { "accountId": "...", "name": "...", "entityId": "...",
      "nativeCurrency": "INR", "nativeBalanceMinor": 0, "baseBalanceMinor": 0 }
  ],
  "debt": {                     // M2-5 summary (active debts), converted to base
    "totalOutstandingMinor": 0,
    "totalMonthlyPaymentMinor": 0,
    "weightedAvgRatePct": 0,
    "debtCount": 0,
    "byType": [ { "type": "home_loan", "outstandingMinor": 0 } ]
  },
  "cashflowSummary": {          // M2-4 summary for `cashflowPeriod` (base currency)
    "period": "2026-07",
    "incomeMinor": 0, "expenseMinor": 0, "netMinor": 0, "savingsRate": 0,
    "byCategory": [ { "category": "housing", "amountMinor": 0 } ]
  },
  "budgetSummary": {            // M2-4 budget vs actual for the period (nullable if none)
    "period": "2026-07", "exists": true,
    "totalBudgetMinor": 0, "totalSpentMinor": 0, "overTotal": false
  },
  "assetAllocation": [          // asset-class % of assets (core allocationFromValues)
    { "assetClass": "equity", "baseValueMinor": 0, "pct": 0 }
  ],
  "currencyExposure": [         // gross exposure by native currency (assets+liabilities)
    { "currency": "USD", "baseValueMinor": 0, "pct": 0 }
  ],
  "householdEquity": {          // net worth ‚àí debt-not-in-liability-accounts (reconciled, ¬ßnote)
    "netWorthMinor": 0, "totalDebtMinor": 0, "reconciledEquityMinor": 0
  },
  "entityHoldings": [           // per-entity roll-up (assets, liabilities, debt, net)
    { "entityId": "...", "assetsMinor": 0, "liabilitiesMinor": 0,
      "debtOutstandingMinor": 0, "netMinor": 0 }
  ],
  "relationships": {            // structural context (ids only; PII stays out of the payload)
    "memberCount": 0, "entityCount": 0,
    "entityIds": ["..."], "accountIds": ["..."]
  }
}
```

> **Reconciliation note (ADR-012).** `netWorth.liabilitiesMinor` stays the **M2-3 account-based** figure so net
> worth is unchanged and backward-compatible. `debt` is the **M2-5 detailed ledger**. `householdEquity`
> reconciles them explicitly (`netWorthMinor`, `totalDebtMinor`, and a `reconciledEquityMinor` that avoids
> double-counting a debt already represented as a liability account). This is the one place the two liability
> views are unified ‚Äî exactly the M2-6 responsibility ADR-011 anticipated.

## 4. Snapshot Immutability

Snapshots are **append-only, immutable, read-only** (ADR-004):

- **Append only** ‚Äî capture inserts a new row with the next `snapshotVersion`. There is **no `UPDATE`** path
  and **no `DELETE`** in the service (hard delete only cascades if the household is removed).
- **Immutable** ‚Äî `payload`, `checksum`, and the envelope are frozen at capture. The `checksum` makes accidental
  or malicious mutation detectable.
- **Read only** ‚Äî consumers **read** snapshots; they never write through them. A correction is a **new** snapshot
  (optionally marking the prior one `status = void` via a new capture event, never an edit).
- **No recomputation** ‚Äî reading a historical snapshot returns its **stored** payload verbatim. The service
  **never** re-derives a past snapshot from current tables; live data has drifted and the snapshot is the
  record. (Only the **`/current` composed view** ‚Äî an explicitly non-persisted convenience ‚Äî reflects live data;
  it is clearly labelled and is never confused with a stored snapshot.)

## 5. Snapshot Creation Rules

A snapshot is created by an explicit capture. Triggers (`generatedBy`):

- **Manual** ‚Äî an advisor/owner captures "now" (`POST ‚Ä¶/financial-snapshot`). **Built in M2-6.**
- **Scheduled** ‚Äî **month-end / quarter-end / year-end** cadence, run by the **M0 job worker** (deferred, like
  M2-8). The contract is ready: the worker calls the same service with `generatedBy = scheduled`. The `period`
  fields make month/quarter/year captures self-describing.
- **Material financial event** ‚Äî a large transaction, a new/closed debt, a big balance change may enqueue a
  capture (`generatedBy = event`). Hook points exist (M2-4/M2-5 mutations); wiring the trigger is deferred to the
  M6 event bus. M2-6 does **not** auto-capture on every write (that would be noise) ‚Äî capture is deliberate.

Idempotency: capture is **not** deduplicated by default (two manual captures a second apart are two valid
snapshots). Scheduled captures will be keyed by period to avoid duplicates (worker concern).

---

## 6. Recalculation Rules

- **Recalculates immediately (live, on read):** only the **`GET ‚Ä¶/financial-snapshot/current`** composed view ‚Äî
  it re-invokes M2-2‚Ä¶M2-5 against live data every call and is **never stored**. This is the "what would a snapshot
  look like right now" preview.
- **Never recalculates:** every **persisted** `FinancialSnapshot`. Once captured, its payload is frozen forever.
  Changing an account, transaction, debt, budget, or FX rate **has no effect** on any existing snapshot. New
  facts are reflected only by capturing a **new** snapshot.
- **Consumers never recompute the kernel.** Downstream modules and AI read snapshots; they do not re-aggregate
  M2-2‚Ä¶M2-5 (¬ß9, ¬ß11). The one allowed live path is the `/current` preview above.

---

## 7. Historical Timeline

- **Timeline** ‚Äî `GET ‚Ä¶/financial-snapshot/timeline` returns snapshots oldest‚Üínewest (indexed on
  `(householdId, capturedAt)`), each a compact envelope + headline figures for trend charts.
- **Monthly / quarterly / annual** ‚Äî every snapshot's `cashflowSummary.period` and `capturedAt` let a consumer
  bucket the timeline by month/quarter/year without recomputation. Scheduled month/quarter/year-end captures
  (¬ß5) produce clean periodic series.
- **Point-in-time restore (read)** ‚Äî `GET ‚Ä¶/financial-snapshot/:id` (or `?asOf=<timestamp>` ‚Üí the latest
  `active` snapshot at/before that instant) returns the **frozen** position as it was. "Restore" here is a
  **read** ‚Äî reconstructing the historical view ‚Äî **not** a write that mutates current state. Writing state back
  to a past position is explicitly out of scope (and would violate immutability).

## 8. Financial Snapshot Version Contract

The whole point of a "contract". Three independent versions travel with every snapshot (¬ß2):

- **Schema Version** (`schemaVersion`, payload shape). Starts at **1**.
  - **Migration strategy:** **additive only** ‚Äî new payload fields may be added under the **same** version if
    strictly optional; a **breaking** change (rename/remove/retype a field) bumps `schemaVersion` to _N+1_ and
    **new** snapshots are written at _N+1_. **Old snapshots are never rewritten** ‚Äî they keep their `schemaVersion`
    and shape.
  - **Backward compatibility:** a consumer written for _N_ must keep reading _N_ snapshots forever. The service
    exposes the version so consumers can branch. A consumer that only understands _N_ ignores _N+1_ (or upgrades).
  - **Deprecation rules:** a schema version is **deprecated** (announced here + in `PROJECT_MEMORY`) but **not
    deleted** while any stored snapshot uses it. Removal requires a data migration that **re-captures or
    up-converts** old snapshots into a new immutable row (never an in-place edit) ‚Äî a deliberate, documented
    event, not a silent change.
  - **Upgrade strategy:** an **up-converter** (pure function `upgrade(payload, from, to)`) can present an old
    payload at a newer shape **on read** for consumers that want only the latest shape ‚Äî without mutating storage.
    M2-6 ships `schemaVersion = 1` and the up-converter registry stub (identity for v1).
- **Engine Version** (`engineVersion`) ‚Äî records the composing-logic version for reproducibility/debugging; does
  not gate consumers.
- **FX Version** (`fxVersion`) ‚Äî records the rate set; a rate change never alters a past snapshot (rates are
  frozen into the converted figures at capture).

## 9. Performance

- **Snapshots are the read model.** Consumers read **one** indexed row instead of fanning out to four engines +
  FX per request. A dashboard/score/report is a single `findUnique`/`findMany`, not an O(accounts+txns+debts)
  recompute.
- **Reuse, don't duplicate.** Capture **composes** the existing services:
  - **M2-2** `HouseholdAccountsService` ‚Äî accounts (assets/liabilities).
  - **M2-3** `HouseholdNetWorthService` ‚Äî consolidated net worth (+ optionally links the latest `NetWorthSnapshot`).
  - **M2-4** `HouseholdCashflowService.summary` / budget ‚Äî cashflow & budget summaries.
  - **M2-5** `HouseholdDebtService.summary` ‚Äî debt.
  - **`@lcos/core`** ‚Äî `convertMinor`, `summarizeCashflow`, `summarizeDebt`, `computeNetWorth`,
    `allocationFromValues`. **No new aggregation math is invented; none is duplicated.**
- **Cost:** capture is O(household size) **once**; every subsequent read is O(1). Timeline is an indexed range
  read. FX is O(1) static lookups. Heavy/scheduled capture runs off the request path (M0 worker). Payload JSON is
  bounded by household size; large books can paginate `assets`/`liabilities` in a future schema version.

## 10. Extension Points

Future modules **plug into snapshots without redesign** ‚Äî they consume the payload (and add their **own**
household-scoped storage), never modify the kernel. New needs are met by **additive** payload fields (¬ß8).

| Module | How it consumes the snapshot |
| --- | --- |
| **Retirement planning** | net worth + savingsRate + allocation as the starting state for projections. |
| **Estate planning** | entity holdings + relationships + household equity as the estate base. |
| **Insurance analysis** | liabilities + income (cashflow) + dependents (relationships) ‚Üí coverage gap. |
| **Goal planning** | net worth + savingsRate + allocation ‚Üí fundability. |
| **Tax planning** | income/expense mix + debt (deductible interest flag, future) + entity holdings. |
| **Family Office dashboard** | reads the latest snapshot directly ‚Äî the headline read model. |
| **Financial Health Score** | a pure function of one snapshot (solvency, savingsRate, debt ratio, diversification). |
| **Forecasting / Monte Carlo / What-if** | seed simulations from a snapshot's frozen state; scenarios never mutate it. |
| **Risk engine** | currency exposure + allocation + leverage (debt/assets). |
| **Asset-allocation / Rebalancing engine** | `assetAllocation` + `currencyExposure` ‚Üí drift & trades. |
| **AI Wealth Advisor** | reads snapshots only (¬ß11). |
| **Document Vault / Estate documents / Trust structures** | attach documents to a snapshot id / entity; snapshot provides the financial context. |
| **Private equity / Real estate / Business ownership / Alternative assets** | new `assetClass`/account types flow through `assets`/`assetAllocation` with **no kernel change**; illiquid valuations become additive payload fields in a future `schemaVersion`. |

**Design rule:** a consumer that needs a field the payload lacks proposes an **additive** `schemaVersion` bump
here ‚Äî it does **not** reach around the kernel into raw tables.

## 11. AI Contract

- **AI reads Financial Snapshots ‚Äî never raw transactional tables.** Every AI feature (coach, second opinion,
  advisor, insights) is grounded on a **specific snapshot id** (or the latest `active`). It must not query
  `Account`/`Transaction`/`Debt` directly, and must not re-aggregate ‚Äî the snapshot is the sole financial input.
- **Why:** reproducibility (an AI answer cites the snapshot it used and can be re-derived), safety (a frozen,
  validated, checksum-verified input ‚Äî no half-written ledger), tenant isolation (one scoped row, not broad table
  scans), and cost (bounded context, not unbounded transaction history).
- **Shape:** AI receives the payload (or a projected subset) + the envelope (so it can state "as of `capturedAt`,
  `schemaVersion` N"). Grounding is explicit and logged. The AI layer treats the snapshot as **read-only** and
  never writes back through it.
- **Enforcement:** AI services depend only on `FinancialSnapshotService` (read methods), not on M2-2‚Ä¶M2-5 repos
  ‚Äî a structural guarantee, documented here and enforced by module boundaries.

## 12. API Contract

Stable, versioned, household-scoped under `HouseholdScopeGuard`. Additive-only; **no breaking changes** to an
existing route/shape (a breaking need becomes a new `schemaVersion` in the payload, or a new versioned route).

| Method & path | Purpose | Roles |
| --- | --- | --- |
| `POST /households/:id/financial-snapshot` | Capture an immutable **household-level** snapshot (`generatedBy = manual`). Body: optional `cashflowPeriod` (defaults to current month). `entityId` is **reserved** (v1 captures are household-level; see ¬ß2 note). | OWNER/ADVISOR/SUPPORT |
| `GET /households/:id/financial-snapshot/current` | **Live composed** preview (not stored). | any in-scope member |
| `GET /households/:id/financial-snapshot/latest` | The latest `active` stored snapshot. | any in-scope member |
| `GET /households/:id/financial-snapshot/timeline` | Envelopes + headline figures, oldest‚Üínewest. | any in-scope member |
| `GET /households/:id/financial-snapshot/:snapshotId` | A specific stored snapshot (full payload). | any in-scope member |

- **Versioning:** the response always includes `schemaVersion`/`engineVersion`/`fxVersion`. Consumers pin to
  `schemaVersion`. The route base can be versioned (`/v1/‚Ä¶`) if a future contract ever needs a parallel shape;
  v1 is implied for M2-6.
- **Stability guarantee:** fields are only added; existing fields keep meaning and type; enums only gain values.
  Anything else is a new version.

## 13. Security

- **RBAC:** reads for any in-scope member (incl. `ANALYST`, read-only); **capture** is a write, gated to
  `OWNER/ADVISOR/SUPPORT` (matches M2-2‚Ä¶M2-5). No new roles (no drift).
- **Tenant isolation:** every route is `HouseholdScopeGuard`-gated (household ‚àà firm ‚àß in caller scope);
  out-of-scope ‚Üí **404**. The snapshot row carries `firmId` + `householdId`; the new table gets **RLS lockdown**
  (Supabase PostgREST closed), like every M2 table.
- **Audit trail:** capture writes an `AuditLog` (`household.financial-snapshot.capture`, `{ firmId, householdId,
  snapshotId }`) via `AuditService` ‚Äî append-only (ADR-005). Reads are not audited (consistent with M2-3/4/5).
- **Encryption / PII:** the payload stores **ids and amounts only** ‚Äî **no names/taxIds** (PII stays in the
  encrypted source tables, ADR-006). `relationships` carries counts + ids, never plaintext identities.
  `checksum` provides tamper-evidence at rest.

## 14. Acceptance Criteria

- `POST ‚Ä¶/financial-snapshot` captures an **immutable** row whose payload equals the composed M2-2‚Ä¶M2-5 view at
  capture, with a correct `checksum`, `snapshotVersion` incrementing per household, and `schemaVersion = 1`.
- `GET ‚Ä¶/current` returns the **live** composed view (never persisted); `GET ‚Ä¶/latest`, `‚Ä¶/:id`, `‚Ä¶/timeline`
  return **frozen** stored snapshots verbatim (no recomputation).
- **Reproducibility:** after mutating accounts/debts/cashflow, a previously captured snapshot is **byte-identical**
  (same payload + checksum) ‚Äî proven by an e2e test.
- **Multi-currency:** all figures are consolidated in the household base currency, asserted against the same
  `@lcos/core` FX used by M2-3/4/5.
- **No duplication:** the service composes M2-2‚Ä¶M2-5 services; it introduces **no new aggregation math** and does
  not read raw tables to re-aggregate what an engine already computes.
- **Reconciliation:** `householdEquity` reconciles M2-3 net worth and M2-5 debt without double-counting
  (ADR-012), asserted by a test.
- **Security:** outsider ‚Üí 404; analyst may read but not capture (403); capture is audited; new table has RLS.
- **Versioning:** every response exposes `schemaVersion`/`engineVersion`/`fxVersion`; an up-converter stub exists
  (identity for v1).
- **Verification:** `migrate reset` clean + **no drift**; build 3/3; lint 4/4; `@lcos/core` green; API e2e green
  (new snapshot suite added); Vercel preview healthy. `main` stays deployable.

## 15. Design review & hardening (STEP 2)

Weaknesses found in the first draft and how they are resolved:

1. **Checksum must be deterministic.** A SHA-256 over `JSON.stringify` is only reproducible if key order is
   stable. **Resolution:** the payload is hashed via a **canonical serializer** (recursively sorted keys) in a
   single shared `@lcos/core`/service helper (`checksumPayload`), used identically at capture and at
   verification. The reproducibility test (¬ß14) locks this in.
2. **`snapshotVersion` race.** Two concurrent captures could compute the same ordinal. **Resolution:**
   `snapshotVersion` is a best-effort **ordinal** (count within the household at capture); the authoritative
   ordering is `capturedAt` + `id`. No unique constraint is placed on it (Postgres treats `NULL` entityIds as
   distinct, which would make a composite unique brittle), so a rare duplicate ordinal is harmless, not a
   failure. Documented, not over-engineered.
3. **BigInt in JSON.** Minor-unit amounts are stored as JS **numbers** (safe < 2^53 ‚âà ‚Çπ90 trillion; far beyond a
   household). Composition serializes `BigInt ‚Üí Number` at the boundary (as M2-2‚Ä¶M2-5 already do). A future
   schema version can switch to strings if books ever approach the limit ‚Äî additive.
4. **`entityId` over-promised.** The first draft implied per-entity capture. **Resolution:** v1 captures are
   **household-level** (`entityId = null`); the envelope field is **reserved** and per-entity detail already
   lives in `entityHoldings`. No half-built per-entity path ships (¬ß2, ¬ß12).
5. **Liability double-count.** Net worth (M2-3, account-based) and debt (M2-5, ledger) both describe liabilities.
   **Resolution:** `netWorth` stays account-based (unchanged, backward-compatible); `householdEquity` is the one
   place that reconciles the two (**ADR-012**); the contract states this explicitly so no consumer double-counts.
6. **Schema evolution safety.** Without a version gate, an additive change could still surprise consumers.
   **Resolution:** `schemaVersion` + an **up-converter registry** (identity for v1) + explicit
   additive/deprecation rules (¬ß8) make evolution safe and non-breaking by construction.
7. **`/current` vs stored confusion.** A live preview that looked like a snapshot would corrupt trust.
   **Resolution:** `/current` is a **distinct, non-persisted** route, never assigned an id/checksum, and labelled
   as live everywhere (¬ß4, ¬ß6, ¬ß12).

## Schema change (additive ‚Äî ADR-010)

- New enums `SnapshotGeneratedBy` (`manual|scheduled|event|migration`), `FinancialSnapshotStatus`
  (`active|superseded|void`).
- New table **`FinancialSnapshot`** ‚Äî the envelope columns (¬ß2) + `payload Json` + `schemaVersion`/`engineVersion`
  /`fxVersion`/`snapshotVersion` + `checksum` + `provenance Json`, with **RLS lockdown** and indexes
  `(householdId, capturedAt)`, `(householdId, snapshotVersion)`, `(firmId)`, `(entityId)`. Back-relation
  `Household.financialSnapshots`. No reshaping of existing models; M2-2‚Ä¶M2-5 untouched.
